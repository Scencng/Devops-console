# MongoDB 在一般业务中的常见用法

## 1. MongoDB 适合什么场景

MongoDB 是文档型数据库，核心对象是 `document`。它比较适合：

- 数据结构不完全固定，经常新增字段
- 一条记录里包含嵌套对象和数组
- 更强调按对象整体读取，而不是复杂 Join
- 需要快速迭代模型、先落业务再逐步收敛结构

常见业务场景：

- 用户画像、账户配置、偏好设置
- 订单快照、审批流记录、任务执行结果
- 日志、事件、审计记录
- 运维平台里的资产元数据、作业记录、告警详情、变更历史

## 2. 一般怎么组织数据库、集合、文档

典型做法是按业务域划分，而不是“想到什么就建什么”：

- 一个数据库承载一个系统、租户或环境
- 一个集合承载一个业务实体或一类事件
- 一个文档代表一条完整业务对象

示例：

- `devops.assets`
- `devops.jobs`
- `devops.audit_logs`
- `billing.orders`

## 3. 业务里最常见的使用方式

### 以对象为中心读写

MongoDB 更常见的做法是把经常一起读取的数据放到一个文档中，减少额外查询。

例如一个工单文档里可以直接带：

- 工单基础信息
- 当前负责人快照
- 当前状态
- 评论列表
- 审批记录摘要

### 允许字段逐步演进

MongoDB 允许同一集合里的文档字段不完全一致，所以常见做法是：

- 前期快速上线
- 中期通过应用层校验、约定文档、JSON Schema 逐步收敛
- 后期对关键字段建立索引和约束

### 用索引保证查询性能

业务里真正决定 MongoDB 是否稳定可用的，不是 CRUD，而是索引设计。

常见索引场景：

- 按 `user_id` 查询用户数据
- 按 `status + created_at` 查询任务列表
- 按业务单号精确定位
- 按时间范围查询日志和审计记录

## 4. 业务开发里最容易踩的坑

- 把 MongoDB 当成随便塞 JSON 的仓库
- 同一含义出现多个字段名
- 时间字段格式不统一
- ObjectId、字符串、数字混用
- 没有索引就直接上线

建议：

- 关键字段命名统一
- 时间统一使用 `Date`
- 主键和关联键类型保持一致
- 为高频过滤和排序字段建立索引

## 5. 站在当前项目角度，后续应该优先做什么

如果这个模块要并入运维平台，更合理的定位是“MongoDB 运维台”：

- 连接 MongoDB 实例
- 浏览数据库、集合、文档
- 受控执行 CRUD
- 查看索引与实例状态
- 记录审计日志
- 对危险操作做权限控制和确认

因此后续优先级应是：

1. 多连接管理
2. 登录接入与 RBAC
3. 危险操作审计
4. 索引与运维信息展示
5. 聚合、导入导出、批量操作
